## What this skill does

When a developer says something like "Build me an approval tracker app" or "/skill Build me an expense management app", this skill:

1. Plans the project (using gstack skill if available, otherwise creates docs/PLAN.md)
2. Scaffolds a Vite + React 18 + TypeScript (strict) + Tailwind CSS v4 project
3. Interactively discovers Power Platform connection IDs by guiding the user through pac CLI commands
4. Designs the UI (using ui-ux-pro-max skill if available, otherwise follows Tailwind-first business app patterns)
5. Implements features with a backend abstraction layer that falls back to localStorage for local dev
6. Guides deployment via pac code push through Dev/Test/Prod environments
7. Generates governance docs, technical design docs, and deployment guides

## Non-negotiable technical decisions

These are hard-won lessons from production Power Apps Code App development:

1. **React 18 + TypeScript strict + Vite 6 (port 3000) + Tailwind CSS v4 + Zustand**
2. **`import.meta.env.VITE_*` ONLY in src/config.ts** — everywhere else imports constants from config.ts
3. **Timeout + localStorage fallback pattern** — the Power Apps SDK bridge doesn't respond at localhost, so every data service call must use Promise.race with a 6-second timeout and fall back to localStorage
4. **NEVER check `window.powerAppsBridge`** — it is ALWAYS undefined. The PA player uses an internal DefaultPowerAppsBridge via postMessage. Detection must be reactive (try → timeout → fallback)
5. **NEVER hand-edit src/generated/ or .power/** — these are generated by pac code add-data-source
6. **pac code add-data-source needs the FULL connection ID** (e.g., shared-sharepointonl-8fd7e2cd-...), NOT the short UUID. Users must run `pac connection list` to get it
7. **Vite substitutes .env at BUILD time** — changing .env after build has no effect, must rebuild
8. **Generated dataSourceName is lowercase** — pac lowercases the table name in generated code
9. **SP_CONNECTOR_ID is always shared_sharepointonline** on commercial M365 — never changes across environments
10. **Both SharePoint and Dataverse backends supported** — determined by VITE_BACKEND_TYPE env var

## Backend service pattern (critical — include in blueprint.md)

The dataService.ts must implement:
- Reactive backend state: getBackend(), onBackendChange() — so UI components can show which backend is active
- withTimeout() wrapper using Promise.race with 6000ms timeout
- CRUD operations (getItems, createItem, updateItem, deleteItem) that try the real backend first, catch timeout, fall back to localStorage
- localStorage fallback using a single key (app_data_v1) with JSON serialization
- BackendBadge component that subscribes to backend state changes

## Connection discovery flow (include in connection-discovery.md)

Step-by-step guide for users who don't know their connection IDs:
1. Verify pac CLI: pac --version (if not found, check VS Code extension path)
2. Auth: pac auth create --environment <ID> (if unknown, guide to make.powerapps.com → Settings → Session details)
3. List connections: pac connection list → find SharePoint/Dataverse row → extract full connection ID
4. If no connection exists, guide through make.powerapps.com → Connections → New → SharePoint/Dataverse
5. Store discovered values in .env and .connection-refs.local
